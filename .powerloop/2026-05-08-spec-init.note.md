# Standing Instructions for Execute SubAgents

Every Execute SubAgent prompt MUST include the following requirements verbatim:

1. **Structure must come from spec-framework, not from REFERENCE.md.** Read `/spec:spec-framework` if uncertain. Section headers must come from spec-framework elements (Intent: Purpose/Users/Impacts/Success criteria/Non-goals; Design: System boundary/User journeys/Interfaces/Presenter/Behaviors/Error scenarios; Consistency: Terminology/Pattern/Form/Contract; Implementation Standards: Architecture/Design Patterns/Testing Style/Code Organization). REFERENCE.md is source material only — its chapter names (Wire Contract, Wire Codec, Guest Binary, Host Ext + Ruby API, Build & Quality, Lessons Encoded, etc.) MUST NOT appear as headers, sub-headers, or feature names in SPEC.md.
2. **System boundary uses Boundary types** (Responsibility / Interaction / Control) per spec-framework, not free-form prose.
3. **Behaviors use State + Operation → Result form.** Error scenarios are listed explicitly per behavior.
4. **User journeys use Context → Action → Outcome form.**
5. **4-step workflow is mandatory**: `/spec:spec-write` → `/spec:spec-review` → `/spec:spec-write` (refinement pass addressing review findings) → `/git:commit`. Do NOT stop after review without applying the refinement pass.
6. **Auto mode active** — do not enter plan mode, do not ask the user questions for routine decisions.
7. **Append-only by default**: SPEC.md already exists; new sections are appended unless the SubAgent is explicitly told to rewrite an existing section.
8. **Constrain design, open implementation** — capture WHAT/WHY, not HOW. No language-specific syntax, type annotations, or method signatures unless they are part of an external contract.

# Standing Instructions for Review / Sample SubAgents

The dispatcher MUST NOT decompose review criteria into a custom checklist inside the prompt. Doing so causes drift: each blind scanner ends up applying a slightly different rubric, and Stage 2 verifiers reach different verdicts even on identical content. The skill (`/spec:spec-review`) is the single source of truth for what counts as a defect.

Required prompt shape for every Review/Sample dispatch:

1. **Goal sentence** — the project goal (single line, identical across all dispatches).
2. **Item under review** — the item name as it appears in the Progress Table.
3. **Target** — file path + line range or heading anchor (e.g., "`SPEC.md` from `### Wire Codec` heading through just before `### Naming Principles`").
4. **Instruction** — "Invoke the `/spec:spec-review` skill on the target section. Apply the skill's own rubric and reporting format; do not improvise additional criteria."
5. **Blinding rule** — "You receive ONLY the goal, item name, and target. You do NOT have prior verdicts, prior findings, the Progress Table, the Log, this item's review history, or signals that the section was previously fixed. Reach your conclusion from the section content alone."
6. **Verdict format** — Stage 1: `PASS` / `SUSPECT` with 1–5 short aspect labels (no line numbers, no quotes, no findings). Stage 2: `PASS` / `FAIL` with concrete location + issue + fix when FAIL. Stage 2 receives the aspect labels only (no specific findings from Stage 1).
7. **Length** — Stage 1: ≤ 80 words. Stage 2: ≤ 250 words.

Fixers (NOT blind): receive full Stage 2 findings verbatim, plus the instruction "Invoke `/spec:spec-write` to address each finding, then run `/git:commit` (or `git commit` via Bash) on `SPEC.md` only — do NOT commit `.powerloop/`."

**Do NOT enumerate criteria like "no internal class names", "no rationale prose", "no Wasm trap", etc., in the prompt.** Those belong to the skill's rubric. Listing them in the prompt biases the scanner toward those specific patterns and causes inconsistent coverage across dispatches.
